# 多专家多轮评审系统

> Hermes 作为 AI 大脑，调度多专家对代码/创意作品进行 3 轮结构化评审，生成可执行的修改约束。

## 系统架构

```
                   ┌──────────────────────────────┐
                   │     用户提交评审材料           │
                   │   (代码/小说/剧本/分镜/设计稿)   │
                   └──────────────┬───────────────┘
                                  ▼
              Phase 1 ── 主题分析 + 专家选型
              │  自动匹配 / huashu-nuwa 蒸馏
              │
              Phase 2 ── R1 独立评审 (deepseek-v4-pro)
              │  每位专家独立输出意见
              │
              Phase 3 ── R2 交叉评审 (deepseek-v4-pro)
              │  看到他人意见后回应/修订
              │
              Phase 4 ── R3 统一意见 (deepseek-v4-pro)
              │  消除分歧 → 加权投票 → 最终结论
              │
              Phase 5 ── 生成评审记录
              │  完整文档 + 修改约束
              │
              ⚡ 执行方式：
                 主流程 → Hermes 原生 + deepseek-v4-pro
                 Fallback → OpenClaude（容器内）
                 最终 Fallback → Kimi 网页版（浏览器自动化）
```

## 执行流程

### Phase 0: 需求收集

收到评审请求后，先收集以下信息：

| 项目 | 说明 |
|------|------|
| **材料类型** | 代码 / 小说 / 剧本 / 分镜头 / 设计稿 / 其他 |
| **评审材料** | 文件路径或全文内容 |
| **评审焦点** | 用户关注的特定方向（可选） |
| **专家数量** | 默认 3 位（可调整 2-5） |
| **已有专家偏好** | 用户指定某几位专家（可选） |

确认后进入 Phase 1。

---

### Phase 1: 主题分析 + 专家选型

#### 1.1 材料分析

先用 **deepseek-v4-flash**（快速、低成本）分析材料：

```bash
# Hermes 直接分析（无需调用外部工具）
分析评审材料的以下维度：
1. 材料类型（代码/小说/剧本/分镜/设计稿）
2. 技术栈/题材/风格
3. 需要哪些专业视角（技术/叙事/视觉/商业/用户）
4. 材料规模和复杂度
```

输出：`材料特征表`，指导专家匹配。

#### 1.2 专家匹配

**强制席位要求：**

| 席位 | 要求 | 来源 | 说明 |
|------|------|------|------|
| **席位① 领域专家** | 材料类型匹配度最高的专家 | perspectives skill 扫描 | 代码→费曼/芒格/塔勒布，剧本→陈思诚/刘和平，设计→乔布斯等 |
| **席位② 市场/投资专家** | 擅长行业分析、风险判断、商业模式评估 | 优先：munger-perspective、naval-perspective、paul-graham-perspective、trump-perspective | 评估可落地性、市场价值、投入产出比 |
| **席位③ 逻辑/批判专家** | 逻辑严谨性、盲点扫描、极端情况检查 | 优先：feynman-perspective、taleb-perspective、ilya-sutskever-perspective | 找逻辑漏洞、反例、边缘条件 |
| **席位④ 补充专家** | 根据材料特征补充不同视角（可选） | perspectives skill 取剩余匹配度最高的 | 增加视角多样性 |
| **席位⑤ 补充专家** | 同上（可选） | 同上 | 尽力凑满 5 位 |

**匹配步骤：**

1. 扫描 `skills/` 下所有 `*-perspective/` 目录
2. 读取每个 SKILL.md 的 description 和心智模型
3. 按评审类型计算匹配度（基于 description 中的关键词、触发词、心智模型适用场景）
4. 按三类席位分别匹配：
   - **领域专家**：取匹配度最高的 → 席位①
   - **市场/投资专家**：从 munger/naval/paul-graham/trump 等中取匹配度最高的 → 席位②
   - **逻辑专家**：从 feynman/taleb/ilya 等中取匹配度最高的 → 席位③
   - **补充专家**：从剩余专家中取匹配度最高的 1-2 位 → 席位④⑤
5. 确保三个席位**不重叠**（同一位专家不能同时占两个席位）
6. 若某类席位无可用匹配 → 用 huashu-nuwa 快速蒸馏一位

**按评审类型的默认席位分配：**

| 评审类型 | 席位① 领域专家 | 席位② 市场/投资 | 席位③ 逻辑/批判 |
|---------|---------------|----------------|----------------|
| **代码/技术** | feynman / taleb | munger（成本效益） | feynman（逻辑）或筛选另一位逻辑型 |
| **剧本/小说/叙事** | chensicheng / liu-heping / ninghao / haiyan | hongguo-editor / shortdrama-veteran（市场判断） | taleb（黑天鹅检验）或 feynman（结构严谨性） |
| **分镜头/视觉** | zhangyimou / wangjing / ninghao | chensicheng（票房商业） | munger（成本合理性） |
| **设计稿/产品** | steve-jobs / zhang-yiming | naval（商业模式） | feynman（用户逻辑） |
| **商业计划/战略** | paul-graham / elon-musk | munger / taleb（风险） | feynman（逻辑验证） |
| **短剧/短视频** | shortdrama-veteran / hongguo-editor / chensicheng | 同上 | taleb（风险检验） |

**匹配完成后输出专家配置表：**

```
┌──────┬──────────────┬──────────────┬────────┬─────────────────┐
│ 席位 │ 专家         │ 核心视角      │ 权重   │ 入选理由         │
├──────┼──────────────┼──────────────┼────────┼─────────────────┤
│ ①   │ 费曼         │ 逻辑严谨性   │ 0.30   │ 代码审查核心     │
│ ②   │ 芒格         │ 成本效益分析 │ 0.25   │ 商业价值判断     │
│ ③   │ 塔勒布       │ 极端情况检验 │ 0.25   │ 边界条件+错误处理 │
│ ④   │ 张一鸣       │ 产品视角     │ 0.10   │ 补充产品可用性   │
│ ⑤   │ —            │ —            │ —      │ 未匹配到第5位    │
├──────┼──────────────┼──────────────┼────────┼─────────────────┤
│      │ 合计         │              │ 1.00   │ 3位专家          │
└──────┴──────────────┴──────────────┴────────┴─────────────────┘
```

**权重计算规则：**
- 基础权重：席位① 0.30-0.35 / 席位② 0.25-0.30 / 席位③ 0.25-0.30 / 席位④ 0.10-0.15 / 席位⑤ 0.05-0.10
- 席位②③④⑤的权重从席位①的基础权重中分配
- 专家越少，每位专家权重越高（3位时约 0.30-0.35/位，5位时 0.20-0.25/位）
- 归一化：所有专家权重求和调整为 1.00

#### 1.3 蒸馏新专家（可选）

如果现有专家库不覆盖某些评审维度，用 **huashu-nuwa** 蒸馏新专家：

```bash
# 触发条件：
# 1. 用户指定了某个人物但不存在
# 2. 评审主题需要特定领域专家
# 3. 用户说「再找一个XX领域的专家」

# 执行蒸馏（deepseek-v4-flash 用于调研，快速完成）
启用 huashu-nuwa skill：
  输入：人名/主题
  输出：新的 perspective skill
  模型：deepseek-v4-flash（蒸馏用轻量即可）
  约束：快速生成（非深度调研），只保留 3-5 个核心心智模型
```

**蒸馏产出物**：临时 skill 文件，评审完成后可保留或删除。

#### 1.4 产出专家配置表

```
┌──────────────┬──────────────┬────────┬──────────┐
│ 专家         │ 核心视角      │ 权重   │ 入选理由  │
├──────────────┼──────────────┼────────┼──────────┤
│ 费曼         │ 逻辑严谨性   │ 0.35   │ 代码审查核心 │
│ 芒格         │ 逆向扫描盲区 │ 0.30   │ 边界条件   │
│ 塔勒布       │ 极端情况     │ 0.35   │ 错误处理   │
├──────────────┼──────────────┼────────┼──────────┤
│ 合计         │              │ 1.00   │          │
└──────────────┴──────────────┴────────┴──────────┘
```

**权重计算规则：**
- 基础权重：专家匹配度（0.3-0.5）
- 加分：专家在该评审类型中的专精度（+0.05-0.15）
- 减分：与另一位专家的视角重叠度超过 70%（-0.1）
- 归一化：所有专家权重求和调整为 1.0

**⚡ CHECKPOINT：展示专家配置表，用户确认后再进入 Phase 2。**

### Phase 2: R1 独立评审

> 模型：deepseek-v4-pro（评审重要性，使用重模型）

#### 2.1 构造每位专家的评审 Prompt

为每位专家构造个性化 prompt，包含：
1. 该专家的 SKILL.md 内容（思维框架）
2. 评审材料
3. 评审模板

```markdown
## 评审 Prompt 模板

你正在以 [专家名] 的身份进行评审。

### 你的思维框架
[SKILL.md 核心内容摘要——心智模型、决策启发式]

### 评审材料
[用户提交的全文]

### 评审要求
请按照你的思维框架，逐项输出：

#### 1. 整体印象（100字内）
- 第一反应：你觉得这个作品/代码怎么样？
- 最亮眼的地方是什么？

#### 2. 具体评审（按你的视角逐条列出）
每条包含：
- 📍 位置/行号（如适用）
- 🔍 发现的问题/亮点
- ⚠️ 严重程度：严重 / 中等 / 建议
- 💡 修改建议

#### 3. 核心结论（3-5条）
从你的视角看，这个材料需要解决的最关键的 3-5 个问题。

#### 4. 评分（满分 10 分）
- 技术/逻辑：X/10
- 创意/独特性：X/10
- 完整性：X/10
- 整体：X/10
```

#### 2.2 执行评审

```bash
# Hermes 直接调用 deepseek-v4-pro
# 每位专家作为一个独立会话执行
# 使用技能上下文注入机制（注入 superpowers 规则）

方案 A（主方案）：Hermes 直接评审
  - 用 Hermes 自身能力 + deepseek-v4-pro 模型
  - 加载对应 expert 的 SKILL.md
  - 按评审 prompt 模板输出

方案 B（Fallback）：OpenClaude 执行
  openclaude -p "以 [专家名] 的身份评审以下材料：[材料]
  参考以下思维框架：[SKILL.md 摘要]
  按模板输出..." \
  --provider anthropic \
  --model claude-sonnet-4-6 \
  --append-system-prompt "这是重要评审任务，请确保：1) 输出结构化 2) 每条意见标注严重程度 3) 给出具体修改建议" \
  --allowedTools "Read" \
  --max-turns 3
```

#### 2.3 收集 R1 结果

将所有专家的评审意见收集为一个文档：

```
│         R1 独立评审汇总                       │
```

### Phase 3: R2 交叉评审

> 模型：deepseek-v4-pro

#### 3.1 匿名化 R1 意见

将 R1 结果中的专家名字替换为「专家A」「专家B」「专家C」，避免权威效应。

#### 3.2 交叉反馈

将其他专家的匿名意见发给每位专家：

```markdown
## 交叉评审 Prompt 模板

你是 [专家名]。

你已完成第一轮评审。现在请阅读其他专家的意见：

### 其他专家意见摘要

专家A认为：
- [意见1]
- [意见2]

专家B认为：
- [意见3]
- [意见4]

专家C认为：
- [意见5]
- [意见6]

### 你的任务

请逐条回应：

1. **同意/补充**：哪些意见你同意？有什么可以补充的？
2. **质疑/分歧**：哪些意见你不同意？为什么？
3. **修订你的意见**：看完他人意见后，你的立场是否有变化？
   - 如果变化：新的意见是什么？为什么改变？
   - 如果不变：为什么坚持？有什么更充分的理由？

4. **最终评分**（如有变化）：更新你的评分
```

#### 3.3 收集 R2 结果

```

│         R2 交叉评审汇总                       │
├─────────────────────────────────────────────┤
│                                              │

│  └── 修订：评分从8→7                        │
│                                              │
│  专家B：                                     │
│  └── ...                                    │
```

### Phase 4: R3 统一意见 + 加权投票

> 模型：deepseek-v4-pro

#### 4.1 分歧识别

汇总 R2 结果，识别所有分歧点：

| 分歧点 | 专家A | 专家B | 专家C |
|--------|-------|-------|-------|
| 问题1（严重程度） | 严重 | 中等 | 严重 |
| 问题2（是否需要修改） | 需要 | 不需要 | 需要 |
| 整体评分 | 7 | 8 | 6 |

#### 4.2 统一讨论

对每个分歧点进行统一讨论：

```markdown
## 统一讨论 Prompt 模板

你是评审协调员。

以下问题存在分歧：

### 分歧点1：[描述]
- 专家A（权重W1）立场：...
- 专家B（权重W2）立场：...
- 专家C（权重W3）立场：...

请模拟三位专家就这个问题进行讨论：
1. 专家A阐述理由
2. 专家B回应
3. 专家C补充或提出折衷方案

目标：找到共识或缩小分歧。
```

#### 4.3 加权投票

对讨论后仍存在的分歧，执行加权投票：

```
分歧点：是否将核心模块重构为异步？
├── 专家A（权重0.35）：✅ 需要重构（异步化是趋势）
├── 专家B（权重0.30）：❌ 不需要（现有方案够用）
├── 专家C（权重0.35）：✅ 需要（但建议分阶段）
│
加权结果：0.35 + 0.35 = 0.70 赞成
        0.30 反对
结论：✅ 通过（赞成 > 50%）
```

**票型输出：**

| 分歧点 | 投票结果 | 加权比分 | 最终决定 |
|--------|---------|---------|---------|
| 核心重构 | ✅ 通过 | 70:30 | 需要重构，分阶段执行 |
| 接口设计 | ❌ 未通过 | 40:60 | 保持现有接口 |
| 评分差异 | 统一 | R2修正后一致 | 整体评分 7.5 |

### Phase 5: 生成评审记录

#### 5.1 完整评审文档

生成结构化评审记录 markdown 文件，保存到 `review-records/` 目录：

```markdown
# 评审记录：[材料名称]

## 元信息
- 评审日期：YYYY-MM-DD
- 材料类型：代码/剧本/设计稿
- 评审专家：[名单及权重]
- 执行模型：deepseek-v4-pro (Hermes原生)

## 专家配置
| 专家 | 视角 | 权重 | 入选理由 |
|------|------|------|---------|
| ...  | ...  | ...  | ...     |

## R1：独立评审意见
[各专家完整意见]

## R2：交叉评审意见
[各专家回应和修订]

## R3：统一讨论记录
[讨论过程和投票结果]

## 最终结论
[结论摘要]

## 修改约束（优先级排序）
1. **P0 必改**：[描述]
2. **P1 建议改**：[描述]
3. **P2 可考虑**：[描述]

## 保留意见
[尚未消除的分歧]

## 评审记录索引
- 记录编号：REV-YYYYMMDD-XXX
- 本记录作为后续修改的输入约束
```

#### 5.2 输出

| 输出项 | 格式 | 用途 |
|--------|------|------|

## 模型选择策略

| 阶段 | 主方案 | Fallback 1 | Fallback 2 |
|------|--------|-----------|-----------|
| 材料分析 | deepseek-v4-flash | — | — |
| 专家搜索/蒸馏 | deepseek-v4-flash | huashu-nuwa | — |
| R1 独立评审 | **deepseek-v4-pro** | OpenClaude (容器内) | Kimi 网页版 |
| R2 交叉评审 | **deepseek-v4-pro** | OpenClaude (容器内) | Kimi 网页版 |
| R3 统一投票 | **deepseek-v4-pro** | OpenClaude (容器内) | Kimi 网页版 |
| 评审记录生成 | deepseek-v4-flash | — | — |

**Fallback 链说明：**

1. **Primary**: Hermes 直接调用 deepseek-v4-pro（最快、最可控）
2. **Fallback 1**: OpenClaude 通过 `--provider anthropic --model claude-sonnet-4-6` 执行（需要容器内可用）
3. **Fallback 2**: Kimi 网页版浏览器自动化（最慢，仅当以上都不可用时）

### Kimi 网页版 Fallback 操作流程

```bash
# 仅当 primary 和 fallback 1 都不可用时启用
# 通过浏览器自动化操作 Kimi 免费网页版

1. browser_navigate("https://kimi.moonshot.cn")
2. browser_type(输入框, "以[专家名]的身份评审以下材料...")
3. browser_click(发送按钮)
4. browser_vision(question="读取回复内容")
5. 复制结果到评审文档
6. 重复以上过程（一轮 3 专家 × 3 轮 = 9 次交互）
```

**⚠️ Kimi 网页版限制：**
- 每次对话有字数上限
- 可能需要验证码
- 速度慢（每次需等待 10-30 秒生成）
- 不支持结构化输出格式控制
- 只做最终 fallback

## Skill 上下文注入

Hermes 调用评审功能时，**自动加载并注入**以下 superpowers skill 规则：

| 阶段 | 注入的 Skill | 规则 |
|------|-------------|------|
| 评审全程 | using-superpowers | 坚持流程、不做跳跃 |
| 输出结论前 | verification-before-completion | 评审记录完整后再声称完成 |
| 分歧严重时 | systematic-debugging | 4 阶段理解分歧本质后再投票 |

注入方式：
- Hermes 原生执行：直接遵守规则
- OpenClaude 执行：`--append-system-prompt "规则: ..."`
- Reasonix 执行：嵌入任务描述首段

## 注意事项

1. **评审质量 vs 速度**：deepseek-v4-pro 费时但质量高，估算每轮每专家约 30-60 秒
2. **3 专家 × 3 轮 ≈ 9 次调用**：完整评审约 5-15 分钟
3. **专家库不足时**：优先从 perspectives skill 匹配，不足时调 huashu-nuwa 快速蒸馏
4. **分歧不强制统一**：如果某位专家坚持己见且理由充分，保留到「保留意见」
5. **权重透明**：投票权重在 Phase 1 公示，用户可调整
6. **评审记录必须保存**：作为后续修改的输入约束，不可跳过
7. **超级权重**：用户可以行使总仲裁权，指定某项意见为最终决定
8. **Kimi 网页版是最后手段**：先走 Hermes 原生 + deepseek-v4-pro
